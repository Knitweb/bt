# BT Sovereign Distribution Plan

BT must not depend on GitHub, GitHub Pages, npm, PyPI, or any single DNS name to
remain distributable. GitHub is useful as a public mirror and issue tracker, but it
is not the source of authority for the project.

## Threat Model

The distribution design assumes that any one of these can fail or become hostile:

- a GitHub organisation, repository, release, issue, or Pages site is removed;
- Microsoft, a US legal order, or a platform policy blocks repository access;
- a DNS name, CDN, package registry, or web host disappears;
- a maintainer loses one workstation;
- one mirror publishes stale or altered code.

The design does not claim that every jurisdiction can be bypassed. The goal is that
the code and release identity can survive the loss of any single platform.

## Authority Rule

A valid BT release is identified by:

1. the Git commit hash;
2. an optional signed Git tag;
3. a source bundle SHA-256 hash;
4. a manifest that lists mirrors, bundle hashes, and restore commands.

The canonical object is the Git object graph plus signatures and hashes. A website
or forge URL is only a retrieval route.

## Distribution Layers

### L1: Full Local Clones

Every maintainer and serious operator should keep a full clone, not only a ZIP
download. A full clone can publish, verify, branch, and restore the repository.

Commands:

```bash
git clone https://github.com/Knitweb/bt.git
git -C bt fsck --strict
git -C bt log --show-signature -1
```

### L2: Mirror Forges

BT should be mirrored to independently operated Git remotes:

- Knitweb-owned Forgejo on the TransIP/5mart hosting side;
- a second European community forge such as Codeberg or another Forgejo instance;
- a generic bare SSH mirror on a host controlled by Knitweb;
- optional GitLab or SourceHut mirror for convenience.

Mirror setup:

```bash
git remote add knitweb-forgejo ssh://git@forge.5mart.ml/knitweb/bt.git
git remote add bare-backup ssh://git@backup.example.org/knitweb/bt.git
git push --mirror knitweb-forgejo
git push --mirror bare-backup
```

Mirror policy:

- push all refs with `git push --mirror`;
- publish the latest manifest on every mirror;
- treat GitHub as one mirror among many;
- never hard-code GitHub as the only recovery path in docs or scripts.

### L3: P2P Code Collaboration

BT should support P2P distribution through Radicle or a similar Git-native network.
This gives the repository an identity that is not controlled by one web forge.

Operational target:

- run at least two Radicle seed nodes controlled by different operators;
- publish the Radicle repository ID in the release manifest;
- accept patches by Git remote, patch file, email, or Radicle patch flow.

### L4: Content-Addressed Release Artifacts

Each release should include a Git bundle and manifest:

```bash
python scripts/make_source_bundle.py
git bundle verify dist/bt-<commit>.bundle
git clone dist/bt-<commit>.bundle bt-restored
```

Publish the bundle and manifest through multiple routes:

- normal web download on each mirror;
- IPFS CID with at least three independent pins;
- BitTorrent with web seeds;
- offline operator copies on encrypted storage.

IPFS should be treated as content addressing and transport, not as a magic
availability guarantee. A CID remains retrievable only when peers keep providing the
content.

### L5: Public Archival

Use Software Heritage for long-term public archival:

- request archive snapshots after meaningful releases;
- record Software Heritage persistent identifiers in the manifest;
- keep these identifiers in `docs/` so future operators can find the source even if
  all normal forges are gone.

### L6: Emergency Recovery

Every major release should have an emergency kit:

- `bt-<commit>.bundle`;
- `bt-<commit>.manifest.json`;
- SHA-256 hash printed in the release notes;
- maintainer public keys;
- a short restore runbook.

Recovery:

```bash
sha256sum bt-<commit>.bundle
git bundle verify bt-<commit>.bundle
git clone bt-<commit>.bundle bt
git -C bt fsck --strict
```

## Release Checklist

Before calling a BT release sovereign-distributable:

- [ ] all tests pass from a clean clone;
- [ ] the release commit or tag is signed by a known maintainer;
- [ ] `scripts/make_source_bundle.py` produced a bundle and manifest;
- [ ] the bundle hash is published outside GitHub;
- [ ] at least two non-GitHub Git mirrors have the same commit;
- [ ] at least one P2P route is announced;
- [ ] the release is submitted to Software Heritage or another public archive;
- [ ] the website links to non-GitHub recovery instructions.

## Current BT Gaps

- No maintained non-GitHub mirror is configured in this repository yet.
- Release tags are not enforced as signed tags yet.
- IPFS, BitTorrent, Radicle, and Software Heritage identifiers are not generated by
  CI yet.
- The GitHub Pages site is still the main public web surface.

## Source Notes

- Git is explicitly designed for distributed workflows where developers can act as
  both nodes and hubs: https://git-scm.com/book/en/v2/Distributed-Git-Distributed-Workflows
- Git bundles are self-contained Git object archives for offline transfer:
  https://git-scm.com/docs/git-bundle
- Radicle is a Git-based P2P code collaboration network:
  https://radicle.dev/
- Forgejo supports self-hosted Git forge operation and repository mirrors:
  https://forgejo.org/docs/latest/
- IPFS content addressing identifies data by content-derived CIDs:
  https://docs.ipfs.tech/concepts/content-addressing/
- Software Heritage persistent identifiers can name archived source artifacts:
  https://docs.softwareheritage.org/devel/swh-model/persistent-identifiers.html
